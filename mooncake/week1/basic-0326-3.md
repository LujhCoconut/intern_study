# SGLang + Mooncake 缓存架构核心概念详解

本文档总结关于 SGLang + Mooncake 缓存架构的三个核心问题：
1. 与通用计算缓存层次结构的本质区别
2. Token 数量、Token ID 大小、KV Cache 大小的关系
3. 变长序列场景下的预取策略

---

## 一、与通用计算缓存的本质区别

### 1.1 通用计算缓存模型

```
┌─────────────────────────────────────────────────────────────┐
│                    通用计算缓存模型（CPU/GPU）                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   L1 Cache (SRAM)                                          │
│   ├── 由硬件管理（对用户透明）                                │
│   ├── 缓存行固定大小（64B/128B）                             │
│   ├── 硬件自动维护一致性（MESI 协议）                         │
│   └── 任意地址都可缓存                                       │
│           │                                                 │
│           ▼                                                 │
│   L2/L3 Cache → 主内存                                       │
│   ├── 硬件/OS 管理                                           │
│   ├── 统一地址空间                                           │
│   └── 自动一致性                                             │
│                                                             │
│   核心特点：                                                  │
│   - 缓存单位：固定大小（缓存行）                               │
│   - 一致性：硬件强一致性（MESI 协议）                          │
│   - 管理：对用户完全透明                                       │
│   - 访问模式：通用随机读写                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 SGLang + Mooncake 缓存模型

```
┌─────────────────────────────────────────────────────────────┐
│                  SGLang + Mooncake 三级缓存模型               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   L1 (GPU 显存)                    L2 (CPU 内存)            │
│   ┌─────────────────┐              ┌─────────────────┐      │
│   │ TreeNode.value  │◄────────────►│ TreeNode.host_  │      │
│   │  (GPU indices)  │   D2H/H2D    │    value        │      │
│   └─────────────────┘   显式拷贝    │  (CPU indices)  │      │
│           │                        └─────────────────┘      │
│           │                              │                  │
│           │                              │ RDMA/网络         │
│           │                              ▼                  │
│           │                        ┌─────────────────┐      │
│           │                        │  L3 (Mooncake)  │      │
│           │                        │  分布式存储      │      │
│           │                        │                 │      │
│           └──────────────────────►│  - 无固定位置    │      │
│              通过 hash 查询         │  - 最终一致性   │      │
│                                   │  - 按需拉取     │      │
│                                   └─────────────────┘      │
│                                                             │
│   核心特点：                                                  │
│   - 缓存单位：可变长度（token 序列）                           │
│   - 一致性：软件管理的弱一致性（最终一致）                      │
│   - 管理：显式管理（HiCacheController）                       │
│   - 访问模式：写一次读多次（WORM）                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心差异对比表

| 特性 | 通用计算缓存 | SGLang + Mooncake |
|------|-------------|-------------------|
| **管理主体** | 硬件/OS | 软件（HiCacheController） |
| **缓存单位** | 固定（缓存行/页） | 可变（token 序列） |
| **一致性** | 硬件强一致性（MESI） | 软件弱一致性（最终一致） |
| **数据可变性** | 频繁修改 | 写入后不可变（immutable） |
| **地址映射** | 统一物理/虚拟地址 | 逻辑索引（RadixTree） |
| **对用户透明性** | 完全透明 | 显式管理 |
| **并发控制** | 硬件锁/缓存一致性 | 软件引用计数（lock_ref） |
| **预取策略** | 硬件预取 | 显式异步预取（后台线程） |

### 1.4 为什么可以放松一致性？

```
┌─────────────────────────────────────────────────────────────┐
│              为什么 KV Cache 可以不需要强一致性？              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 写一次读多次（WORM）                                     │
│     - KV Cache 生成后不会被修改                               │
│     - 只有 insert，没有 update                               │
│     - 示例："北京" 的 KV 一旦生成，永远是这个值               │
│                                                             │
│  2. 前缀共享，互不干扰                                        │
│     - "北京景点" 和 "北京美食" 共享 "北京"                   │
│     - 但各自有自己的叶子节点，不会同时修改同一节点              │
│                                                             │
│  3. 确定性计算                                               │
│     - 给定输入 token，KV Cache 输出是确定的                    │
│     - 不存在 race condition                                  │
│                                                             │
│  4. 最终一致性足够                                           │
│     - L3 备份是异步的                                         │
│     - 即使不同节点看到的数据略有延迟，也没关系                  │
│     - 因为数据一旦写入就不会改变                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、Token 相关大小关系澄清

这是最容易混淆的概念，需要明确区分三个不同的"长度/大小"：

### 2.1 三个概念对比

```
┌─────────────────────────────────────────────────────────────┐
│                    Token 相关大小概念                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 1. Token ID 存储大小                                   │ │
│  │    ────────────────────────                           │ │
│  │    - "北京" → Token ID: 42                            │ │
│  │    - "有"   → Token ID: 156                           │ │
│  │    - 存储类型：int32（4 字节）                          │ │
│  │    - ✅ 固定大小！                                     │ │
│  │                                                       │ │
│  │    类比：图书馆的索书号，每个都是固定格式的编码            │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 2. Token 数量（序列长度）                               │ │
│  │    ────────────────────────                           │ │
│  │    - "你好" → 2 tokens                                │ │
│  │    - "详细解释量子力学的基本原理..." → 1000+ tokens    │ │
│  │    - 取决于输入文本的长度                               │ │
│  │    - ❌ 可变！                                         │ │
│  │                                                       │ │
│  │    类比：借书的数量，有人借 1 本，有人借 10 本            │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 3. 单个 Token 的 KV Cache 大小                        │ │
│  │    ───────────────────────────────                    │ │
│  │    - 在同模型内，每个 token 的 KV 数据大小固定           │ │
│  │    - DeepSeek MLA: ~7KB per token                     │ │
│  │    - 标准 MHA: 数十 KB ~ 数百 KB per token            │ │
│  │    - 计算公式：2 × num_layers × hidden_dim × dtype    │ │
│  │    - ✅ 在同模型内固定！                               │ │
│  │                                                       │ │
│  │    类比：每本书的档案卡大小是固定的                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 为什么 TreeNode 需要管理变长数据？

```
┌─────────────────────────────────────────────────────────────┐
│              为什么 TreeNode 管理的是变长数据？                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  虽然单个 token 的 KV 大小固定，但：                           │
│                                                             │
│  1. 序列包含的 token 数量不同                                  │
│     ┌────────────────────────────────────────────┐          │
│     │ 序列 A: "北京" → 2 tokens                  │          │
│     │ KV Cache 大小 = 2 × 7KB = 14 KB           │          │
│     │                                             │          │
│     │ 序列 B: "北京有哪些景点" → 6 tokens        │          │
│     │ KV Cache 大小 = 6 × 7KB = 42 KB           │          │
│     └────────────────────────────────────────────┘          │
│                                                             │
│  2. RadixTree 中节点的覆盖范围不同                            │
│     ┌────────────────────────────────────────────┐          │
│     │           [Root]                           │          │
│     │             │                              │          │
│     │             ▼                              │          │
│     │         [北京] ← 覆盖 2 tokens             │          │
│     │          /   \                             │          │
│     │         ▼     ▼                            │          │
│     │    [有哪些] [的景点]                       │          │
│     │    3 tokens  3 tokens                     │          │
│     └────────────────────────────────────────────┘          │
│                                                             │
│  因此：                                                       │
│  - TreeNode.value 存储的是一段连续的 token 索引               │
│  - 这段索引的长度是可变的                                     │
│  - 需要 RadixTree 这种变长数据结构来管理                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 页式管理：解决变长存储的方案

```
┌─────────────────────────────────────────────────────────────┐
│              页式管理（Paging）解决变长问题                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：变长序列如何高效存储和管理？                            │
│                                                             │
│  解决方案：按固定大小的 page 分配内存                          │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 示例：page_size = 64 tokens                          │   │
│  │                                                      │   │
│  │ 输入序列："北京有哪些好玩的景点"（8 tokens）          │   │
│  │                                                      │   │
│  │ 分配方式：                                            │   │
│  │ ┌─────────────────────────────────────────────────┐ │   │
│  │ │ Page 0: [北京有哪些好玩的景点 _ _ _ ... _]      │ │   │
│  │ │         8 tokens 使用，56 slots 空闲            │ │   │
│  │ │         (内部碎片)                              │ │   │
│  │ └─────────────────────────────────────────────────┘ │   │
│  │                                                      │   │
│  │ 读取方式：                                            │   │
│  │ - 不管实际用了多少，都按 64 tokens 的块处理           │   │
│  │ - 预取、驱逐都以 page 为单位                          │   │
│  │ - 简化内存管理                                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  代码体现（memory_pool.py）：                                │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ class MHATokenToKVPool:                             │   │
│  │     def __init__(self, size, page_size, ...):       │   │
│  │         self.page_size = page_size  # 通常 64       │   │
│  │         # 分配连续的大块内存                           │   │
│  │         self.k_buffer = torch.empty(                │   │
│  │             (num_layers, size, head_num, head_dim)  │   │
│  │         )                                           │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  权衡：                                                        │
│  - 优点：简化分配、便于批量传输、利于预取                      │
│  - 缺点：内部碎片（最后一页可能不满）                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、变长序列的预取策略

### 3.1 预取的核心挑战

```
┌─────────────────────────────────────────────────────────────┐
│              变长序列预取的核心挑战                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：如果每个请求长度都不一样，如何有效预取？                  │
│                                                             │
│  关键洞察：LLM 请求的"可预测性"                               │
│                                                             │
│  1. 前缀共享模式（最重要）                                     │
│     ┌─────────────────────────────────────────────────┐      │
│     │                                                 │      │
│     │  Request 1: "请解释量子力学"                     │      │
│     │  Request 2: "请解释量子力学的基本原理"            │      │
│     │  Request 3: "请解释量子力学与相对论的区别"         │      │
│     │                                                 │      │
│     │  共同前缀："请解释量子力学" （6 tokens）           │      │
│     │                                                 │      │
│     │  预取策略：                                       │      │
│     │  - 处理 Request 1 时，把"请解释量子力学"的 KV       │      │
│     │    存到 L2/L3                                     │      │
│     │  - Request 2/3 到来时，发现前缀命中，直接从 L2 加载   │      │
│     │    不需要重复计算！                                 │      │
│     │                                                 │      │
│     └─────────────────────────────────────────────────┘      │
│                                                             │
│  2. Page 粒度对齐（固定预取单位）                              │
│     - 预取不是按 token 粒度，而是按 page 粒度                  │
│     - 序列长度 70 → 对齐到 128（2 pages）                     │
│     - 预取单位是固定的！                                       │
│                                                             │
│  3. RadixTree 的局部性                                        │
│     - 同一分支上的节点有空间局部性                             │
│     - 访问父节点后，很可能访问子节点                            │
│     - 可以预取兄弟节点                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 预取流程详解

```
┌─────────────────────────────────────────────────────────────┐
│                    预取流程详解                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 请求到达                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 新请求："请解释量子力学的基本原理"                    │   │
│  │ Token 序列：[请, 解释, 量子, 力学, 的, 基本, 原理]    │   │
│  └──────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  Step 2: RadixTree 前缀匹配                                  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 查找结果：                                            │   │
│  │ - "请解释量子力学" 已在树中（L1/L2 命中）             │   │
│  │ - "的基本原理" 需要重新计算（未命中）                 │   │
│  │                                                     │   │
│  │ 匹配节点：TreeNode("请解释量子力学")                 │   │
│  │ 剩余 tokens：[的, 基本, 原理]                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  Step 3: 触发预取（异步）                                     │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ # hiradix_cache.py                                   │   │
│  │ def prefetch_from_storage(self, seq):                │   │
│  │     # 计算需要预取的长度（按 page 对齐）               │   │
│  │     prefetch_len = align_to_page(len(remaining))     │   │
│  │                                                      │   │
│  │     # 异步从 L3 加载到 L2                              │   │
│  │     self.cache_controller.prefetch_async(            │   │
│  │         hashes=compute_hashes(tokens[:prefetch_len]),│   │
│  │         dest_buffer=l2_buffer                         │   │
│  │     )                                                │   │
│  └──────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  Step 4: 后台传输 + 前台计算并行                               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 前台：使用已匹配的 KV 继续生成 token                    │   │
│  │       ↓                                             │   │
│  │ 后台：RDMA 从 L3 拉取预取的数据 → L2                  │   │
│  │       ↓                                             │   │
│  │ 当需要新 KV 时，检查发现已在 L2，直接加载到 L1         │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 预取效果对比

```
┌─────────────────────────────────────────────────────────────┐
│                    预取效果对比                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  场景：3 个用户同时询问 LLM                                   │
│                                                             │
│  用户 A: "请解释量子力学"                                     │
│  用户 B: "请解释量子力学的历史"                               │
│  用户 C: "请解释量子力学的数学基础"                            │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 无预取策略（或没有 L2/L3 缓存）                        │   │
│  ├──────────────────────────────────────────────────────┤   │
│  │  A: 计算"请解释量子力学" 100ms                         │   │
│  │        ↓                                             │   │
│  │  B: 计算"请解释量子力学" 100ms （重复计算！）          │   │
│  │        ↓                                             │   │
│  │  C: 计算"请解释量子力学" 100ms （重复计算！）          │   │
│  │                                                      │   │
│  │  总时间: ~300ms                                       │   │
│  │  重复计算了 3 次相同的前缀！                          │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 有 RadixTree + L2/L3 + 预取                           │   │
│  ├──────────────────────────────────────────────────────┤   │
│  │  A: 计算完整序列，KV 存入 L2/L3         100ms         │   │
│  │        ↓                                             │   │
│  │  B: 命中 L2，加载"请解释量子力学"       5ms          │   │
│  │     只需计算"的历史"部分               +20ms         │   │
│  │        ↓                                             │   │
│  │  C: 命中 L2，加载"请解释量子力学"       5ms          │   │
│  │     只需计算"的数学基础"部分           +25ms         │   │
│  │                                                      │   │
│  │  总时间: ~155ms（提升 1.9x）                          │   │
│  │  共同前缀只计算一次！                                  │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  关键洞察：                                                   │
│  - 虽然 B 和 C 的序列长度不同（变长），但前缀是固定的！        │
│  - 预取的是共同前缀的 KV Cache，不是任意长度的随机序列          │
│  - Page 粒度让预取单位固定，简化管理                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、总结

### 4.1 核心要点回顾

```
┌─────────────────────────────────────────────────────────────┐
│                      核心要点回顾                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 缓存架构差异                                              │
│     ┌─────────────────────────────────────────────────┐      │
│     │ • 通用计算：硬件管理、固定行大小、强一致性         │      │
│     │ • SGLang+Mooncake：软件管理、可变序列、弱一致性   │      │
│     │ • 原因：KV Cache 的 immutable + WORM 特性         │      │
│     └─────────────────────────────────────────────────┘      │
│                                                             │
│  2. Token 大小关系（重要！）                                  │
│     ┌─────────────────────────────────────────────────┐      │
│     │ • Token ID 大小：固定（int32 = 4 字节）           │      │
│     │ • Token 数量：可变（取决于输入文本长度）           │      │
│     │ • KV Cache/每 token：在同模型内固定               │      │
│     │ • TreeNode 管理范围：可变（不同节点覆盖不同长度）  │      │
│     └─────────────────────────────────────────────────┘      │
│                                                             │
│  3. 变长序列预取                                              │
│     ┌─────────────────────────────────────────────────┐      │
│     │ • 预取单位：Page（固定大小，如 64 tokens）        │      │
│     │ • 预取依据：前缀共享（RadixTree 匹配）            │      │
│     │ • 执行方式：异步后台线程（与计算并行）            │      │
│     │ • 关键：利用 LLM 输入的前缀局部性                  │      │
│     └─────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 设计哲学

```
┌─────────────────────────────────────────────────────────────┐
│                    设计哲学总结                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  SGLang + Mooncake 的缓存设计是对 LLM 推理特性的深度优化：      │
│                                                             │
│  1. 利用数据特性                                              │
│     - KV Cache 的 immutable → 可以放松一致性                  │
│     - 前缀共享模式 → RadixTree 高效去重                       │
│     - 读多写少 → 异步预取有意义                               │
│                                                             │
│  2. 适应硬件约束                                              │
│     - GPU 显存有限 → 需要 L2/L3 卸载                          │
│     - RDMA 网络可用 → 远程存储访问快                           │
│     - 多机部署 → 需要分布式缓存层                              │
│                                                             │
│  3. 软件定义缓存                                              │
│     - 不依赖硬件缓存一致性                                     │
│     - 显式管理数据流动                                         │
│     - 应用层感知（知道哪些是共同前缀）                          │
│                                                             │
│  这种设计牺牲了通用性（只能用于 KV Cache 场景），                │
│  换取了极致的性能优化。                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
